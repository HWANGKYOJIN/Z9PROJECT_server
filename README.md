## 🏷️ 기획의도

Z9은 광고보다 인플루언서의 경험을 더 신뢰하는 현대인을 위한 공동구매 플랫폼으로, 공동구매의 0부터 9까지 모두 다루겠다는 의미를 이름에 담았다. 2022년 현재 인플루언서가 공동구매(이하 공구)를 개최할 때 제품사 쇼핑몰 혹은 N사의 스토어를 이용한다. 전용 플랫폼이 아니기에 공구가 언제 열릴지, 얼마나 팔리는지,  현황을 알 수 없다. Z9는 단순 URL 공유를 넘어 인플루언서의 개성을 그대로 이어가는 데 목표를 뒀다. 공구 예약 시스템, 다양한 상세페이지, 실시간 판매량 표기 등 여러 기능을 도입한다. 소비자는 더 나은 공동구매 방식을 경험하고, 인플루언서는 깔끔한 공구 진행으로 프로페셔널한 이미지를 얻을 수 있다.

### 👂 팀프로젝트를 통해 배운 대화의 중요성

  백엔드 구성원 중 나를 제외한 나머지 2명이 이미 전공자이거나 2년이상 경험이 있는 팀원으로 구성 되어 있어서, 팀프로젝트 팀장직을 맡아 수행하는 것이 엄청나게 부담이 되었습니다. 기술적으로는 제가 제일 부족했기때문에 팀장으로써 팀원들의 의견을 조율하고, 소통의 장을 만들어주는 것을 우선 목표로 삼았습니다. ERD를 설계할때 Table간에 많은 Join이 DB를 Query할 때 미치는 영향이라던지 git commit rule 정할 때의 효율성 등. 처음인 내가 생각해보지도 못한 문제점들을 제시하는 팀원들을 보며, 나의 지식이 정말 한없이 부족하다는 것을 느끼게 되었습니다. 하지만 팀장으로써 의견을 조율해야 했고 의견을 조율하려면 그 의견이 대립하는 이유를 알아야 하기에 몰랐던 내용을 공부해보고, 각 의견에 대한 장단점을 고려하여 의견제시자에게 자신의 의견과 상대방 의견에 대한 장단점을 인지 시킨 후 다수결로 대립된 의견을 조정하여 문제를 해결하였습니다. 최대한 모든 안건은 각 의견의 장단점을 고려하여 투표를 통해 다수결로 정하는 것을 원칙으로 하였습니다. 프로젝트 초반에는 백엔드와 프론트엔드와 회의도 자주하고, 의견도 공유하면서 전체적인 흐름에 대한 의견공유를 자주하였는데, 프로젝트가 중반쯤 넘어가게 되면서 제가 맡은 업무를 수행하는 것에 급급하여 팀장으로써 전체적인 프로젝트의 진행방향 진행율 체크에 미흡했고,기능 연결 시간이 필요한 프론트엔드와 기능연결하면서 발생하는 오류에 대한 생각을 미쳐못하고 프로젝트가 거의 끝나는 시기까지 API 완성을 계획으로 잡는 오류를 범하였고 프론트엔드와의 전체회의 요청으로 전체적인 일정을 재 조율하였고, 프로젝트를 무사히 마무리 할 수 있었습니다. 프론트엔드에 대한 지식이 상대적으로 부족함에서 온 문제였고, 전체적인 흐름을 아는 것이 또한 중요하다는것을 느꼈습니다. 전체적인 흐름을 공부해볼수 있게 개인프로젝트는 프론트부터 백엔드까지 간단하게 혼자 구현해보는 것을 계획중입니다.

### 🤝 Git을 통한 협업

  간단하게 개인레포지토리에서 git add. git commit, git push 만 하던 저에게 팀프로젝트를 통해 git에 대해 새로운 경험을 할 수 있었습니다. commit시 지켜야 할 규칙, branch명을 지을 때 규칙등을 정하여 팀원들과 업무를 분담하여 작업하고, 각자 작업한 내용들을 합치고, PR을 하면서 merge시 충돌이 나는 케이스, PR했던 내용들이 사라져 작업한 내용을 다 소실된 경험 등 다양한 오류들을 겪을 수 있는 귀중한 경험 이었습니다. 또한 코드리뷰를 통해 다른 팀원들이 작성한 코드를 읽어보며, 어디 까지 작업했고, 어떤 스타일의 코드를 작성하고, 혹시나 틀린부분이 없는지 체크를 해주면서 다른사람의 코드를 리뷰해주는 것이 공부가 되었고, 혹여나 내가 리뷰한 코드에서 오류나 오타가 발생하면 제대로 리뷰하지 못했다는 책임감도 느낄수 있었습니다. 하지만 프로젝트 막바지에는 오류코드를 우선적으로 해결해야 했기 때문에 처음에는 잘지켜졌던 규칙들을 지키지 못하였습니다. 팀프로젝트를 Github를 통해 하면서 느낀점은 가능하면 '업무세분화에 신경쓰자.', '정해진 규칙을 지키자.'였습니다. 팀프로젝트 채택한 브랜치 전략은 그림과 GIT-FLOW를 따랐고, feature 브랜치를 생성하였고, feature 브랜치에서 작업하는 하위 브랜치를 별도로 나누어진 브랜치에서 작업을 하였는데, 먼저 제출한 PR이 나중에 작업할 이슈에 영향을 미칠 시 먼저 제출한 PR이 merge되지 않으면 후 작업도 못하게 된다는 문제가 있었습니다. merge가 실시간으로 계속 이루어지는게 아니기 때문에 이럴때 마다 새로운 이슈를 발생시키지 않고 merge가 되기 전까지 이전 브랜치에서 작업 후 merge가 완료되면 임시저장 후 다시 작업하는 과정을 반복해야만 했습니다. 이를 통해 각각의 업무를 세분화 할 때 서로에게 영향을 상활을 고려하여 업무를 세분화 해야 추후작업에 영향없이 작업을 이어 나갈수 있다는것을 배웠습니다.
  
### 🗂️ 공공데이터포털 활용하기

  프로젝트를 처음 실행하기 전부터 프로젝트시 단순시 유저가 데이터를 입력하면 저장하고 이를 DB에서 꺼내어 보여주는 것보다, 검증할 데이터들이 있으면 오픈API를 최대한 이용 유저가 입력한 데이터를 검증하였으면 좋겠다는 생각을 했습니다. 공공데이터포털에서는 우리 팀프로젝트에서 활용할 수 있는 사업자등록번호 검증을 하였습니다. 셀러로 가입을 하게되면 사업자등록번호를 입력하는 칸이 있는데, 여기에 유저가 사업자 등록번호를 입력하게되면 공공데이터포털에서 국세청에 가입되어있는 사업자번호들과 비교하여 국세청에 가입되어 있지않은 사업자번호를 입력시 사업자번호를 확인해달라는 오류를 반환하도록 구현하였습니다. 두번째로는 유저가 환불 또는 정산을 위해 계좌번호를 등록 할 수 있도록 되어있는데, 아임포트에서 제공하는 API를 활용하여 계좌번호와 예금주가 서로 일치하는지를 검증하였습니다. 입력받은 데이터를 오픈API를 활용해 검증하는 것이 이메일과 같은 입력받은 데이터를 알고리즘을 통한 로직을 통해 검증하는것이 아닌 새로운형태의 검증방법으로 프로젝트에서 다양한 오픈API를 활용해 데이터를 검증해 DB에 보다 깔끔한 데이터를 저장 할 수 있다는점을 배울 수 있었습니다.
