## 🏷️기획의도

 Z9은 광고보다 인플루언서의 경험을 더 신뢰하는 현대인을 위한 공동구매 플랫폼으로, 공동구매의 0부터 9까지 모두 다루겠다는 의미를 이름에 담았습니다.  2022년 현재 인플루언서가 공동구매(이하 공구)를 개최할 때 제품사 쇼핑몰 혹은 N사의 스토어를 이용합니다. 전용 플랫폼이 아니기에 공구가 언제 열릴지, 얼마나 팔리는지,  현황을 알 수 없었습니다.  이러한 불편 사항들을 개선하고자 공구 예약 시스템, 실시간 판매량 표기, 현재 진행하고 있는 공구 등 다양한 기능을 활용하여 사용자는 다양한 공구 정보들을 습득할 수 있습니다.

## ❔ 프로젝트 기간 중 배운점과 느낀점

### 👂  팀프로젝트를 통해 배운 대화의 중요성

 이미 개발 경험이 있는 팀원들 사이에서 팀 프로젝트 팀장직을 맡아 수행하는 것이 부담되었습니다. 팀장으로서 팀원들의 의견을 조율하고, 소통의 장을 만들어주는 것을 우선 목표로 삼았습니다.
 
 
 충돌하는 의견을 결정할 때는 다수결을 통해 해결하였습니다. 팀 프로젝트를 진행하면서 사소한 부분부터 기술적으로 고려해야 하는 것들까지 정해야 할 것이 많았습니다. 기술적으로 내가 모르는 내용들도 있었고, 양해를 구한 후, 따로 공부하여 내용을 숙지 후 다수결을 통해 이견을 조율하였습니다. 기술적으로 중대한 문제가 발생하는 의견이 아닌 이상 모든 안건들의 장단점을 고려하여 투표를 통해 다수결로 의사를 결정하였습니다.  
   
 침묵은 조타수없는 배와 같다. 프로젝트 초반에는 백엔드와 프론트엔드와 회의가 자주 있어 의견을 공유하면서 전체적인 흐름을 파악하는 것에 어려움이 없었습니다. 프로젝트가 중반쯤 넘어가게 되면서 제가 맡은 업무를 수행하는 것에 급급하여 팀장으로서 전체적인 프로젝트의 진행 방향 진행율 체크에 미흡했습니다. API 연결 시간이 필요한 프론트엔드와 API를 연결하면서 발생하는 오류, 발생한 오류를 수정하는 시간 등을 고려하지 못하고, 백엔드 API를 프로젝트가 거의 끝나는 시기까지 완성하는 계획으로 잡는 오류를 범하였습니다. 프론트엔드와의 전체 회의 요청으로 일부기능을 조율하여, 전체적인 일정을 재조율하였고 프로젝트를 무사히 마무리 할수 있었습니다.
프론트엔드에 대한 지식이 상대적으로 부족함에서 온 문제였고, 전체적인 흐름을 아는 것이 또한 중요하다는 것을 느꼈습니다.

### 🤝  Git을 통한 협업

 간단하게 개인 레포지토리에서 git add. git commit, git push만 하던 저에게 팀 프로젝트를 통해 git에 대해 새로운 경험을 할 수 있었습니다. commit시 지켜야 할 규칙, branch 명을 지을 때 규칙 등을 정하여 팀원들과 업무를 분담하여 작업하고, 각자 작업한 내용들을 합치고, PR을 하면서 merge시 충돌이 나는 케이스, PR했던 내용들이 사라져 작업한 내용을 다 소실된 경험 등 다양한 오류들을 겪을 수 있는 귀중한 경험이었습니다.  
  
 또한 코드 리뷰를 통해 다른 팀원들이 작성한 코드를 읽어보며, 어디까지 작업했고, 어떤 스타일의 코드를 작성하고, 혹시나 틀린 부분이 없는지 체크해주면서 다른 사람의 코드를 리뷰해주는 것이 공부가 되었습니다. 혹여나 내가 리뷰한 코드에서 오류나 오타가 발생하면 제대로 리뷰하지 못했다는 책임감도 느낄 수 있었습니다. 하지만 프로젝트 막바지에는 오류코드를 먼저 해결해야 했기 때문에 처음에는 잘 지켜졌던 규칙들을 지키지 못하였습니다.  
  
 팀 프로젝트를 Github를 통해 하면서 느낀 점은 가능하면 '업무 세분화에 신경 쓰자.', '정해진 규칙을 지키자.'였습니다. 팀프로젝트에서 feature-branch에서 작업할때 제일 먼저 특징별로 feature 브랜치를 생성하고, 생성한 feature 브랜치에서 작업하는 하위 브랜치를 별도로 나누어진 브랜치에서 작업을 하였는데, 먼저 제출한 PR이 나중에 작업할 이슈에 영향을 미칠 시 먼저 제출한 PR이 merge되지 않으면 후작업도 못 하게 된다는 문제가 있었습니다. merge가 실시간으로 계속 이루어지는게 아니기 때문에 이럴 때마다 새로운 이슈를 발생시키지 않고 merge가 되기 전까지 이전 브랜치에서 작업 후 merge가 완료되면 임시저장 후 다시 작업하는 과정을 반복해야만 했습니다. 이를 통해 업무 세분화를 할 때 각각의 업무에게 미치는 영향을 고려하여 업무 세분화를 해야 추후 작업에 영향 없이 작업을 이어 나갈 수 있다는 것을 배웠습니다.

### 🗂️  공공데이터포털 활용하기

 프로젝트를 처음 실행하기 전부터 프로젝트 시 단순히 사용자가 데이터를 입력하면 저장하고 이를 DB에서 꺼내어 보여주는 것보다, 검증할 데이터들이 있으면 오픈 API를 최대한 이용 사용자가 입력한 데이터를 검증하였으면 좋겠다는 생각을 했습니다. 
 
 공공데이터포털에서는 우리 팀 프로젝트에서 활용할 수 있는 사업자등록번호 검증을 하였습니다. 판매자로 가입을 하게 되면 사업자등록번호를 입력하는 칸이 있는데, 여기에 사용자가 사업자 등록번호를 입력하게되면 공공데이터포털에서 국세청에 가입되어있는 사업자 번호들과 비교하여 국세청에 가입되어 있지 않은 사업자번호를 입력하면 검증오류를 반환하도록 구현하였습니다. 
 
 두 번째로는 사용자가 환불 또는 정산을 위해 계좌번호를 등록할 수 있게 되어있는데, 아임포트에서 제공하는 API를 활용하여 계좌번호와 예금주가 서로 일치하는지를 검증하였습니다.  
  
 입력받은 데이터를 오픈API를 활용해 검증하는 것이 이메일과 같은 입력받은 데이터를 알고리즘을 통한 로직을 통해 검증하는 것이 아닌 새로운 형태의 검증 방법으로 프로젝트에서 다양한 오픈API를 활용해 데이터를 검증해 DB에 더욱 깔끔한 데이터를 저장할 수 있다는 점을 배울 수 있었습니다.
